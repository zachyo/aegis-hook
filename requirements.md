How to get started
Please post your hook ideas in ⁠#hook-ideas and get feedback or suggestions (or maybe even collaborators)
Find teammates in #form-a-team (you can also work solo, if you prefer)
If you haven’t submitted your Hookathon Project Idea Submission via this form, this should be your top focus before starting work. If you’re stuck on an idea, spend a few minutes reading through what people have shared on discord and scan through the Request for Hooks page for inspiration.
Be sure to submit your progress updates.
Submitting your progress updates:
Submit this form for the Progress Update 1 (1 week into the hookathon)
Submit this form for the Progress Update 2 (2 weeks into the hookathon)
Make sure you use the Project ID you recieved in your initial submission email (should be something like HK-UHI#-####)

Reminder: Only one person from each team needs to submit the update!

Submitting your project via the Final Hookathon Submission form:
Submit your final project here.
Once submitted, your project will appear in the hooks directory: hooks.atrium.academy
You’ll receive an email invite within 1 week of submission for edit access to your individual project page.
(Check for an email with subject \*\*"Atrium is ready to work with you")

Tips for your Hookathon Submission
Things to note while submitting:
Under Submission Type, select “Uniswap Hook Incubator (UHI)”
Select all the partners you’ve integrated so you're considered for their track prizes a. Please do not select any partners that you haven't actually built the integrations for. Explanations for future integrations will not be considered for prizes.
In the Links section, make sure to include:
a. A demo video (no more than 5 minutes). You can use Loom, OBS or any screen capture tools you'd like
b. Your public GitHub repo(s) that contain the hook and any supporting code
For your submission, testnet deployment isn't mandatory. Just make sure that you submit a hook with: unit tests OR a basic frontend that interacts with the hook
Demo Video Tips
You have full creative freedom — just keep it clear, concise, and under 5 minutes.

Here are some formats you can use:

Start with slides explaining your hook, then move into a live demo
Short walk through your code and tests — or if you have a frontend, show us that
Make sure to cover these:

The problem you're solving
How your hook works
How it compares to existing solutions (if relevant)
Most people use either Youtube or Loom to share their demo videos with us, but you can also use Google Drive links, Dropbox links, Vimeo, or upload your video to your project's github repo and send us the link to that. We're flexible on what formats we accept, we just ask that you make sure it's accessible by all the judges- if we can't watch it, we can't grade it.

For some examples of demo videos that have scored highly in the past, check out these:

Introducing Hookupasar: A New Service for Uniswap v4
modl - Modular On Chain Dynamic Logic

Judging
This section is meant to help you ensure you’re planning for the best possible Hookathon outcome and that you submit a hook that meets all the necessary requirements to be eligible for judging and for graduation.

Binary Qualifications
If any of these criteria are not true or present, your submission will not be evaluated in judging and you may not be eligible for alumni status.

Criteria Description Pass/Fail Threshold Notes
Public GitHub Repo GitHub repository is public. Yes Private repos must be cleared with Atrium Academy prior to submission and access provided to Hookathon judges. Private repos that have not been pre-cleared will be disqualified.
Demo/Explainer Video Demo or explainer video (≤5 min) explaining the hook is included in the submission. Yes Without a demo video, your project will not reviewed for judging.
Valid Hook Contains a valid Uniswap v4 hook or directly interfaces with hooks. Yes You must have built a Uniswap v4 hook to be eligible for judging and alumni status. Any exceptions must be submitted to team@atrium.academy and approved prior to the submission deadline.
Written & Workable Code Submission contains functional code created during the Hookathon period. Yes While you may have some code that was written prior to the Hookathon start date, a portion must be newly written during the Hookathon period.
README with Partner Integrations README lists partner integrations used and where they are in code (or states “No partner integrations”). Theoretically integrations or future plans to integrate will not be considered, and should not be mentioned for judging. Yes Your README must clearly outline where you’ve integrated partner tech to be reviewed for any partner prize.
New Code for Returning Teams Returning participants may only submit new code and functionality created since last judged submission. Yes Past code that was reviewed in a previous Hookathon is ineligible for prize consideration.
Originality (No Copied Code) Project must be original; directly copied code from workshops, curriculum, other developers, or past projects disqualifies the submission unless clearly credited. Yes Violations disqualify for prizes, graduation, and alumni perks.
Note: A front-end is not required, but providing one may make judging more straightforward. If no front-end is provided, submission must include sufficient test cases for judges to verify functionality. Projects that have neither tests nor a frontend will not be judged for prizes.

Scored Evaluation (Weighted)
Scored on a 0–5 scale (0 = Poor/Not at all, 5 = Exceptional).

For Hook Submissions that meet the Binary Qualifications listed above, judging will be based on the following categories below.

Criteria Description Score Range Weight (%)
Original Idea Novelty of the concept itself — is the idea behind the hook new to Uniswap or DeFi? 0–5 30%
Unique Execution Distinctiveness of how the hook is built — architecture, integrations, optimizations, UX. 0–5 25%
Impact Potential value to users and/or Uniswap ecosystem, or DeFi. 0–5 20%
Functionality How well the hook works as intended/outlined; robustness and completeness of implementation. 0–5 15%
Presentation Pitch Clarity, persuasiveness, and quality of the video explanation/demo. 0–5 10%
